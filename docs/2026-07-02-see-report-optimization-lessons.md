# SEE Report Optimization Lessons

Date: 2026-07-02

Scope: SEE 25-question thinking portrait report, user-facing report generated by the active `/api/report` portrait flow.

## Current Accepted Output Direction

The user-facing report should read like a polished interpretation report, not a debug trace.

Accepted section order:

1. `核心特质画像`
2. `功能区解读`
3. `成长建议`
4. `数据说明`

Rejected section/content patterns:

- Do not show a `解读依据` section.
- Do not start the report with algorithm or evidence explanation.
- Do not show A/B/C/D count bookkeeping in the final report.
- Do not start a function-area paragraph with visible text like `选项分布为A1、B2、C2，没有D`.
- Do not mention `没有D`; it is mechanically correct but not useful to the reader.
- Do not include meta-explanations such as `面对每个功能区的解读，都遵循同一条推理链...`.
- Do not say `所有结论均来自规则命中` in the report body.

The prompt may use structured evidence internally, but the final report should translate that evidence into natural interpretation.

## Prompt Lessons

Negative examples can leak into model output. If a term should not appear in the user report, avoid placing the exact term in the active prompt, even inside a ban.

This happened with hidden/out-of-scope concepts. The safer pattern is neutral wording:

```text
不要提及当前资料未覆盖的其他测评体系、指标名称或数据项。
```

Avoid unsafe negative wording such as:

```text
报告中不得出现 [hidden term]。
```

because the model may copy the hidden term into `数据说明`.

For deleted content, do not include the deleted sentence as a negative example. Use neutral instruction:

```text
不要添加说明分析方法、规则来源或推导流程的元描述段落。
```

## Validator Lessons

Validator checks must match the current product-facing structure.

For this report, the portrait section check should expect:

- `核心特质`
- `功能区解读`
- `成长建议`
- `数据说明`

The validator should not force user-facing reports to mention internal or out-of-scope metric names merely to satisfy a "data-root" check. For SEE-only reports, `SEE 卡 25 题`, `用户补充字段`, `思维画像`, `功能区`, and natural behavior descriptions are acceptable data grounding.

## Review Checklist For Future Report Systems

Before accepting a report-generation change:

1. Confirm the active endpoint and active prompt path.
2. Check whether secondary prompt files are used by another endpoint; do not assume similarly named templates are active.
3. Run compile checks.
4. Run a minimal validator repro matching the target user-facing structure.
5. Inspect the active prompt string for:
   - forbidden user-facing headings
   - hidden metric names
   - deleted paragraphs included as negative examples
   - mechanical score bookkeeping instructions
6. Generate or inspect a sample report and check that it reads like a user report, not an audit log.
7. Confirm Claude has read the latest whiteboard instruction before waiting for implementation.

## Collaboration Rule Learned

Claude may read only the first screen or first few lines of `AGENT_WHITEBOARD.md`.

For actionable tasks to Claude:

- Put the latest instruction at the absolute top of `AGENT_WHITEBOARD.md`.
- Keep the first instruction block short and explicit.
- After writing the whiteboard, verify via tmux capture that Claude's monitor read it.
- Do not rely only on the `Log:` section.
- Do not use long tmux input messages; they may remain stuck in Claude's input box.

## Applying This To The Next Innate-Trait Report Work

For the upcoming innate-trait report optimization, first identify:

- active endpoint
- active prompt file
- supporting knowledge files
- validator path
- sample output users are reacting to

Then apply the same acceptance lens:

- The first section should be the user's meaningful portrait, not evidence.
- Evidence and computation should support the writing, not be displayed as raw process.
- Internal terms should be exposed only when they are part of the intended user vocabulary.
- Deleted or sensitive concepts should not appear in the prompt as examples.
- The validator should enforce the desired user-facing report, not an old template.
